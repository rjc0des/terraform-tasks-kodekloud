Task 3

Create VPC Using Terraform

Stand up the Nautilus migration network foundation with Terraform

🧩

Problem Statement

Must Have

Nautilus is breaking its AWS migration into small, observable phases. This task wants a single VPC called nautilus-vpc in us-east-1 with a valid IPv4 CIDR block, managed entirely from the Terraform working directory at /home/bob/terraform. Everything must live in /home/bob/terraform/main.tf and the network resource must be created purely through Terraform.

Deliver cloud networking that is repeatable, tag-driven, and easy to consume for later phases without touching the console.

🧠

Approach & Thinking

Strategy

Use aws_vpc to declare the desired CIDR block, name, and region. Locking the configuration inside main.tf keeps the migration phase encapsulated so it can be reviewed and applied independently of later steps.

Alternative would be to reuse a shared VPC, but that violates the incremental isolation requirement. Provisioning a dedicated VPC now avoids dependency collisions down the road and lets us tag it for discovery.

Keeping everything Terraform-native also lets future phases reference the VPC attributes directly through data sources instead of guessing manual IDs.

🛠️

Terraform Solution

Blueprint 
resource "aws_vpc" "nautilus_vpc" {
  cidr_block = "10.0.0.0/16"
  tags = {
    Name = "nautilus-vpc"
  }
}
▶️

Step-by-Step Execution

Steps
  1. Run terraform init in /home/bob/terraform to download the AWS provider (pinned implicitly by the working directory configuration).
  2. Run terraform plan to verify the CIDR, name tag, and region are correct before applying.
  3. Run terraform apply to create the VPC and persist the state for downstream phases.

Required provider: hashicorp/aws configured for us-east-1 with valid credentials. Prerequisites: AWS IAM permissions for ec2:CreateVpc and network tagging, plus write access to /home/bob/terraform.

✔️

Why This Works

Proof

aws_vpc guarantees the CIDR block, tags, and lifecycle are managed by Terraform so the migration phase remains reproducible.

Tagging with Name = "nautilus-vpc" satisfies visibility requirements and lets later automation reference the expected name instead of guessing IDs.

Keeping the logic inside main.tf matches the assessor requirement and ensures we know the exact resource created during review.

📚

Reference to Terraform Docs

Concepts

aws_vpc defines AWS networking boundaries and exposes attributes such as the CIDR block and default tenancy. Terraform automatically tracks it in state, enabling dependency graphs and clean destroy sequences.

Resource tagging through the tags map lets cloud operations find the VPC by name instead of digging through IDs.

⚠️

Common Mistakes

Watch
  • Forgetting to pin the AWS region, which can land the VPC somewhere unexpected and break downstream automation.
  • Omitting the Name tag so auditors or later tasks cannot find the expected VPC.
  • Writing the VPC block in a different .tf file than Main.tf, violating the assignment instructions.
  • Using a /24 CIDR that overlaps existing networks and causing routing conflicts for future migration phases.
🪄

Final Thoughts

Wrap Up

This single Terraform resource produces a self-testsable network foundation that can be referenced by future workloads without manual tweaks, keeping the Nautilus migration predictable.
