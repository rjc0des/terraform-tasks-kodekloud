Task 2

Nautilus Security Group for the Migration Sprint

Capture the new phase by defining the trusted ingress rules for the default VPC

📄

Problem Statement

Must Have

Nautilus is breaking the AWS migration into controlled steps. This piece needs a Terraform-managed security group named nautilus-sg inside the us-east-1 default VPC, anchored in /home/bob/terraform/main.tf. The SG must have a clear description and allow HTTP (80) and SSH (22) from everywhere via explicit ingress rules.

The goal is to tell reviewers that the group exists, why it guards production workloads, and how the migration remains repeatable without poking the console.

🧠

Approach & Thinking

Strategy

Read the default VPC with data "aws_vpc" "default" so the SG attaches automatically, avoiding hard-coded IDs and keeping the stage isolated yet consistent with the existing account.

Define the SG with the mandated name and description, then carve out two entrance rules via aws_security_group_rule resources to keep HTTP and SSH explicit. Everything stays inside main.tf so the migration sprint is auditable and simple to extend.

Locking the provider to us-east-1 keeps the plan predictable, while separating the ingress rules avoids Terraform drift when future engineers tweak ports.

🛠️

Terraform Solution

Blueprint 
data "aws_vpc" "default" {
  default = true
}

resource "aws_security_group" "nautilus_sg" {
  name        = "nautilus-sg"
  description = "Security group for Nautilus App Servers"
  vpc_id      = data.aws_vpc.default.id
  tags = {
    Name = "nautilus-sg"
  }
}

resource "aws_security_group_rule" "allow_http" {
  type              = "ingress"
  from_port         = 80
  to_port           = 80
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.nautilus_sg.id
  description       = "Allow HTTP from anywhere"
}

resource "aws_security_group_rule" "allow_ssh" {
  type              = "ingress"
  from_port         = 22
  to_port           = 22
  protocol          = "tcp"
  cidr_blocks       = ["0.0.0.0/0"]
  security_group_id = aws_security_group.nautilus_sg.id
  description       = "Allow SSH from anywhere"
}
▶️

Step-by-Step Execution

Steps
  1. terraform init inside /home/bob/terraform to download hashicorp/aws and register the default VPC data source.
  2. terraform plan to verify the security group name, description, and both ingress rules target ports 80 and 22 from 0.0.0.0/0.
  3. terraform apply to create the security group in us-east-1 and let the migration state capture the required ingress policy.

Required provider: hashicorp/aws configured for us-east-1. Prerequisites: AWS credentials with ec2:CreateSecurityGroup and ec2:AuthorizeSecurityGroupIngress, plus write access to /home/bob/terraform.

✔️

Why This Works

Proof

data.aws_vpc.default ties the SG to the default network in the account without forcing manual edits, so the resource is immediately usable for the next migration wave.

Splitting the ingress definitions into aws_security_group_rule keeps HTTP and SSH from being intertwined, making future adjustments straightforward and testable.

Keeping everything inside main.tf satisfies the constraint and makes the phase self-contained so reviewers can see exactly what Terraform will touch before downstream workloads rely on it.

📚

Reference to Terraform Docs

Concepts

provider "aws" selects the us-east-1 endpoint and authenticates calls so the plan targets the correct region.

data "aws_vpc" "default" reads the built-in VPC object, letting downstream resources reference its ID without copying it from the console.

aws_security_group defines the logical group, while aws_security_group_rule resources carve out each ingress port with clear descriptions.

⚠️

Common Mistakes

Watch
  • Forgetting to fetch the default VPC first, which results in a security group without any VPC ID and a failed apply.
  • Nesting the ingress rules inside the aws_security_group block, which makes incremental updates and auditing harder.
  • Omitting the Name or description, so reviewers cannot verify the resource intent during checkpoints.
  • Targeting the wrong region or leaving the provider unspecified, which can silently create the SG in another account altogether.
